## 1. 目标

V3.3 不是再做一轮 UI 扩张，而是把 V3.2 的页面骨架收口成“服从当前 repo-native 接口事实”的前端控制台。

本轮唯一目标：

1. 用接口矩阵替换“理应支持”的字段承诺。
2. 让 artifact mode 真正站在 repo-native artifact 上，而不是前端自带聚合 fixture 上。
3. 修掉 mock / artifact 模式隔离与 validation 截断这两个会破坏可信度的问题。

---

## 2. 最高约束

V3.3 的前端 PR 需求、provider contract、页面字段、测试断言，全部必须服从接口矩阵。

强制要求：

1. 不能再写“后端理应支持”“后续接真实后端即可”。
2. 不能把 provider 合成字段伪装成后端原生字段。
3. 不能把 `snapshotRef` / `replayRef` / `validationRef` / `gateRef` 写成天然存在的后端主键。
4. 不能再把 `updatedAt` 当作通用原生字段。
5. 不能再承诺 review timeline、review 时间筛选、稳定 replay_id、原生 validation list 这类当前仓库并不存在的能力。

---

## 3. 当前后端事实

当前 repo 的真实后端面只有：

- `ReplayBundle`
- `IncidentReport`
- `ReleaseAttemptRecord`
- `NarrativeQualityReviewRecord`
- `JsonArtifactStore` 对上述对象的按路径 JSON 读写

当前 repo **没有**：

- 独立 HTTP 后端
- 原生 artifact list/index
- 原生 snapshot artifact
- 原生 replay list / validation list / overview summary / metrics summary
- 带时间元数据的 review list 契约

因此，V3.3 的 artifact mode 必须显式承认：

- 页面能看到的很多面是 `provider 派生面`
- 这些派生面要么来自 `CAT`，要么来自全量遍历原始 artifact
- 任何无法从当前对象稳定推出的字段，都必须从 P0 承诺中删除或降级

## 5. 本轮 P0

### 5.1 P0-1 repo-native artifact ingestion

目标：

- 补一层正式的 `CAT`（catalog/index）产物，作为 repo-native artifact 的中间索引，而不是继续依赖前端手写 fixture。

要求：

- `artifact mode` 的数据源必须改成“原始 artifact + CAT”
- `CAT` 必须明确标注哪些字段是：
  - 原始 artifact 字段
  - catalog 元数据
  - provider 派生字段
- `CAT` 至少覆盖：
  - `ReplayBundle`
  - `IncidentReport`
  - `ReleaseAttemptRecord`
  - `NarrativeQualityReviewRecord`

验收：

- Artifact / Snapshot / Replay / Review / Incident / Release 列表页不再依赖前端手写 fixture
- 文档不再把当前 artifact mode 表述成“已经天然对齐后端”

### 5.2 P0-2 query cache 按 `dataMode` 完全隔离

目标：

- mock / artifact 两种模式下的 query cache、placeholder data、错误回退完全隔离。

要求：

- 所有 query key 必须纳入 `dataMode`
- 不允许 `mock -> artifact` 失败后继续展示 mock 缓存
- `Showing cached ...` 必须只指向当前模式的缓存

验收：

- mock 数据不会污染 artifact 模式
- provider 切换和 query cache 行为一致

### 5.3 P0-3 Validation Center 不允许静默截断

目标：

- Validation Center 看到的结果必须等于当前 provider 可见 replay 集合的完整派生结果。

要求：

- 不允许固定只抓 `page=1&pageSize=50` 的 replay 再拼 validation
- 二选一：
  - provider 层补 `listValidations(...)`
  - 页面层遍历全量 replay 后再聚合
- `timeRange` / `replayOperatorId` 若不是原生稳定字段，必须在文档和 UI 里写成 best-effort

验收：

- 不再存在超过 50 条 replay 时的静默丢数
- validation finding 总量与 provider 可见 replay 集合一致

## 6. 明确不进入本轮 P0 的事项

以下问题保留为后续收口，不阻塞 V3.3：

- artifact detail 中全局 metrics 与 bundle 局部 metrics 的最终语义拆分
- Metrics 页面默认 gate ref 的更细粒度交互

但即便不在本轮完成，也必须满足两条：

1. 文档里要保留风险说明。
2. 页面不能继续用误导性表述伪装这些语义已经正确收口。

---

## 7. 文档与测试要求

V3.3 期间必须同步更新：

- 前端 PR 需求文档
- provider contract 描述
- 页面测试断言
- README / 前端说明文档中对 artifact mode 的表述

统一要求：

- 只写“当前能保证”的字段
- 对派生字段写清来源对象
- 对降级能力写清限制条件

---

## 8. 工程验收

V3.3 交付前至少验证：

- `pytest -q`
- `frontend/npm run test`
- `frontend/npm run coverage`
- `frontend/npm run build`

并补充三类定向验证：

- artifact mode 是否真正消费 `CAT + 原始 artifact`
- mock / artifact 切换是否完全隔离缓存
- Validation Center 是否对大于 50 条 replay 仍然完整

---

## 9. 结论

V3.3 的成功标准不是“页面更多了”，而是“页面字段终于不再超过后端事实”。

如果 V3.2 解决的是“前端能不能跑起来”，那么 V3.3 要先解决的是：

1. 当前页面主键是不是后端真的能支撑
2. 当前页面字段是不是当前对象真的能保证
3. 当前 artifact mode 是不是真的在消费 repo-native artifact

只有这三件事先收口，后面的全局指标、局部指标、复杂跳转语义才值得继续做。
