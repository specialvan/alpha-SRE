# alpha-SRE V3.2 Frontend Review

## 结论

本轮对更新后的 `claude-review/v3/alpha-SRE-frontend-PR.md` 的评审结论仍然是：`request changes`。

和 `V3`、`V3.1` 两轮返审相比，这版文档已经进入收尾阶段。前面几轮里更大的方向性问题基本都已经处理，包括：

- 已明确采用 `Artifact-First / Mock-First`
- 已把顶层 artifact 与 bundle-derived section 区分开
- 已修正 replay 路由主键依赖
- 已补齐 `Artifact Browser` 与 `ReleaseAttemptRecord` 的 provider 覆盖
- 已把 review 页回链从硬保证收敛为 `best-effort`

因此，这一轮不再重复上一轮已经修掉的问题。当前剩余问题主要是 3 个契约细节仍未完全对齐：

1. `Validation Center` 的过滤维度超出了当前 artifact / provider 模型能稳定保证的字段面
2. “所有列表统一分页契约”仍然对部分记录类型承诺了并不存在的过滤字段
3. 工程验收里对 `mock` / `artifact` 双模式的要求被意外放松，和文档前文的 P0 范围不一致

## 总体判断

这版文档已经从“需要重定方向”进入到“需要补最后一层契约一致性”的阶段。

也就是说，现在需要打回 Claude 的，不再是产品方向或信息架构，而是实施口径与验收口径必须彻底自洽：

- 页面过滤项必须和当前真实可取到的数据一致
- 统一列表契约必须允许“按数据源能力降级”，而不是默认所有列表都具备同样字段
- P0 的工程验收必须和 P0 的范围定义保持完全一致，不能留下“做一半也算过”的漏洞

如果这 3 个点不修正，前端在真正进入实现时，仍然会出现：

- provider 层不得不拼装隐含字段
- 某些列表页为了满足验收而前端伪造状态/时间字段
- 最终 PR 在只完成 `mock` 或只完成 `artifact` 其中一种模式时就被误判为达标

## 已确认修正的上一轮问题

以下问题在当前版本中已得到实质处理，不建议再重复打回：

- provider abstraction 已补齐 `Artifact Browser` 与 `ReleaseAttemptRecord`
- artifact 资源模型已区分 top-level artifact 与 bundle-derived section
- review 页回链已明确收敛为引用可解析时的 `best-effort`

## 剩余 Findings

### F1. Validation Center 仍然假定当前模型稳定提供可过滤的时间与操作人字段

对应位置：

- `claude-review/v3/alpha-SRE-frontend-PR.md:261-266`

问题说明：

`Validation Center` 当前要求支持以下过滤维度：

- `failure class`
- `subject id`
- `affected field`
- `time range`
- `replay operator id`

但按当前文档自己定义的 provider 面：

- validation 并不是独立顶层 artifact
- provider 暴露的是 `getValidationForReplay(ref)`
- 并没有一个单独的 `listValidations(...)` 聚合查询面

再结合当前仓库真实模型：

- validation 结果主要来自 replay bundle 内部段
- `replay operator id` 来自 `ReplaySession`
- `ReplayBundle.session` 仍然是可选项

这意味着对 validation 做统一的时间范围和 `replay operator id` 过滤，并不是当前模型天然保证的能力，而是需要：

- provider 回溯 replay bundle
- 尝试从 bundle session 中恢复字段
- 在字段缺失时做降级

影响：

- 当前文档把“可推导的过滤维度”写成了“硬性 P0 过滤维度”
- 实现时 provider 很可能需要额外发明聚合层或补造 metadata
- 一旦某些 bundle 没有 session，页面就会在过滤能力上出现不一致

修订建议：

- 二选一明确化：
  1. 新增 `listValidations(...)` 这类 provider 聚合接口，并定义其返回的 operator/time 元数据来源
  2. 保留当前 provider 面，但把 `time range` 和 `replay operator id` 改成“若源数据可解析则支持，否则降级不可用”

### F2. 统一列表分页契约仍然对部分记录类型过度承诺了时间/状态过滤

对应位置：

- `claude-review/v3/alpha-SRE-frontend-PR.md:461-468`

问题说明：

文档当前写的是：

- 所有列表页必须支持 `page / page_size / total`
- 所有列表页必须支持时间范围过滤
- 所有列表页必须支持状态过滤
- 所有列表页必须支持搜索与排序

这个要求对某些页面是合理的，但对当前仓库中的部分记录类型并不成立。

例如：

- `NarrativeQualityReviewRecord` 没有原生时间字段
- `NarrativeQualityReviewRecord` 没有原生状态字段
- bundle-derived validation section 也不具备统一的 status/time 模型，除非前端或 provider 回推父级 artifact 元信息

所以，这一节目前的问题不是“要求高”，而是“把 only-when-supported 的能力写成了 all-lists must-have 的统一契约”。

影响：

- 前端会被迫为某些记录类型补造并不属于后端 contract 的字段
- 列表页之间的过滤能力表面统一，底层语义却不一致
- 验收时容易出现“形式满足、语义失真”的结果

修订建议：

- 把这节改成分层表述，例如：
  - 所有列表页必须支持 `page / page_size / total`、搜索与排序
  - 时间范围过滤与状态过滤仅在数据源原生提供或 provider 可稳定推导时启用
- 或者明确列一个“字段能力矩阵”，说明不同列表页各自支持哪些过滤维度

### F3. 工程验收现在允许在缺失 `mock` 或 `artifact` 之一时通过 P0

对应位置：

- `claude-review/v3/alpha-SRE-frontend-PR.md:577-578`

问题说明：

前文已经明确：

- `mock` 是 P0 必须支持
- `artifact` 也是 P0 必须支持

但工程验收写成了：

- `artifact provider / mock provider 至少实现其一，并能通过统一接口切换`

这里出现了明显的范围松动：

- 如果只做 `mock provider`，就违背了 `Artifact-First`
- 如果只做 `artifact provider`，就违背了“无后端也能联调”的 mock-first 开发路径

也就是说，这条工程验收给了一个不符合前文范围定义的通过漏洞。

影响：

- PR 可能在只实现一半数据模式的情况下被误判达标
- 最终落地结果会和文档标题宣称的 `Artifact-First / Mock-First` 不一致
- 后续再补另一种模式时，会把本应属于 P0 的工作拖到后续阶段

修订建议：

- 如果 `mock` 与 `artifact` 都是 P0，工程验收应明确写成：
  - `mock provider` 与 `artifact provider` 均已实现
  - 并且都能通过统一 provider 抽象切换
- 如果作者确实想降低范围，那就必须同步修改前文，把其中一种模式降级出 P0

## 建议给 Claude 的修订动作

建议本轮只要求 Claude 做以下三类修订，不必再大改结构：

1. 把 `Validation Center` 的过滤维度改成“provider 明确支持的能力”，或补充 validation 聚合接口
2. 把统一列表契约改成“基础能力 + 按源数据支持启用的高级过滤能力”
3. 把工程验收中的 `artifact provider / mock provider 至少实现其一` 改为与 P0 范围一致的双实现要求

## 最终口径

这版 V3.2 文档已经非常接近可执行输入，剩下的问题不再是方向问题，而是“契约写法是否足够严谨”。

现在建议打回 Claude 的焦点只有这 3 条：

- Validation 过滤项是否与当前模型一致
- 统一列表契约是否允许按数据源能力降级
- P0 双模式是否在工程验收里被完整保住

这 3 个点修掉之后，这份文档基本就可以进入下一阶段实施。
