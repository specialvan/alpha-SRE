# alpha-SRE V3.1 Frontend Review

## 结论

本轮对更新后的 `claude-review/v3/alpha-SRE-frontend-PR.md` 的评审结论仍然是：`request changes`。

和上一轮相比，这版文档已经有实质改进，尤其是以下几个方向已经明显收敛：

- 文档定位从“默认存在可查询 live 后端”收敛为 `Artifact-First / Mock-First`
- replay 详情路由不再强依赖不存在的稳定 `replay_id`
- Metrics 部分已补入 V2.2 narrative-native 指标分组
- Quality Review 页面要求已开始向现有 `NarrativeQualityReviewRecord` 契约收缩

因此，这一轮不再重复上一版已经被修正的大问题。当前剩余问题主要集中在：

1. 文档内部的数据抽象层仍未闭环
2. artifact-first 模式下哪些对象是“顶层 artifact”、哪些只是 bundle 内嵌段，还没有定义清楚
3. 少量验收口径仍然超出了现有 review contract 的可保证范围

## 总体判断

这版文档已经不再是“方向失真”的稿子，而是进入了“设计已基本对齐，但仍缺少少数关键契约闭环”的状态。

也就是说，现在需要打回的重点不再是战略方向，而是要防止前端在进入实现时再次出现两种常见偏差：

- 页面落地依然绕开自己声明的 provider abstraction
- artifact browser 把 bundle 内嵌对象误当成天然可独立寻址的顶层资源

如果这两个点不补清楚，前端虽然可以开始写页面，但数据访问层和资源路由层很快就会出现分叉，后续仍然会返工。

## 已确认修正的上一轮问题

以下问题在本轮文档中已得到实质处理，不建议再作为主 findings 重复打回：

- 已明确采用 `Artifact-First / Mock-First`
- 已把 replay 路由主键改为 `replayRef` / artifact reference 思路
- 已补充 V2.2 narrative-native metrics
- 已在 Quality Review 里显式写出“以现有 record 字段为准”

## 剩余 Findings

### F1. Provider abstraction 仍然无法覆盖两个已声明的 P0 页面

对应位置：

- `claude-review/v3/alpha-SRE-frontend-PR.md:71-85`
- `claude-review/v3/alpha-SRE-frontend-PR.md:142-147`
- `claude-review/v3/alpha-SRE-frontend-PR.md:527-555`

问题说明：

文档已经把 `SreDataProvider` 定义成统一数据访问抽象，并明确要求：

- 页面只依赖抽象层
- 页面不直接依赖数据来源

但当前这个接口里仍然没有：

- `Artifact Browser` 所需的 artifact index / artifact detail 访问方法
- `ReleaseAttemptRecord` 的 list/detail 访问方法

而这两块都已经被文档明确写入 P0：

- `P0.2 Artifact Browser`
- `P0.8 Incidents & Releases`

这会直接导致实现层面出现自相矛盾：

- 一部分页面走 provider
- 一部分页面绕开 provider 自己读 artifact 或拼查询

这会破坏文档自己定义的“页面只依赖抽象层”原则。

影响：

- 数据层会在实现第一阶段就分叉
- `mock / artifact / live` 三模式切换无法做到真正统一
- 后续一旦补 live provider，这两个页面大概率需要单独重构

修订建议：

- 在 `SreDataProvider` 中补齐至少以下方法：
  - `listArtifacts(query)`
  - `getArtifact(ref)`
  - `listReleaseAttempts(query)`
  - `getReleaseAttempt(ref)`
- 或者明确写出：
  - `Artifact Browser` 属于 provider 之外的底层能力
  - 但如果这样写，就必须同步修改“页面只依赖抽象层”的总原则

### F2. Artifact Browser 仍把 bundle 内嵌段当成天然独立 artifact

对应位置：

- `claude-review/v3/alpha-SRE-frontend-PR.md:142-147`

问题说明：

文档当前把以下对象一起表述为 Artifact Browser 的“关键对象”：

- `snapshot`
- `replay`
- `validation`
- `metrics`
- `gate`
- `incident`
- `review`
- `release attempt`

但按当前仓库真实持久化模型：

- 独立持久化的顶层 artifact 主要是：
  - replay bundle
  - incident report
  - release attempt record
  - quality review record
- `snapshot`、`replay`、`validation`、`metrics`、`gate` 主要是 replay bundle 内部段
  - 它们不是天然独立寻址的顶层 artifact
  - 需要由前端或 provider 做 bundle extraction 才能单独展示

所以当前这节虽然已经转向 artifact-first，但仍然缺一层关键定义：

- 哪些是 top-level artifacts
- 哪些是 derived sections
- derived sections 的 reference 如何从 bundle ref 派生

影响：

- Artifact Browser 会把“浏览顶层 artifact”与“展开 bundle 内嵌段”混成一种资源模型
- 前端无法稳定定义：
  - artifact 列表项类型
  - URL 规则
  - detail 页 ref 格式
  - 从 incident/release 跳转到 replay/validation 的方式

修订建议：

- 在文档中显式区分两类对象：
  - Top-level artifact
  - Bundle-derived view
- 至少补一段类似规则：
  - replay bundle 是顶层 artifact
  - `snapshot/replay/validation/metrics/gate` 是从 replay bundle 中提取的派生视图
  - 派生视图使用 `bundleRef + sectionKey` 作为默认 reference

### F3. Review 页回链验收口径仍然超出了当前 review contract

对应位置：

- `claude-review/v3/alpha-SRE-frontend-PR.md:675-676`

问题说明：

文档已经在 `Quality Review` 一节中写出：

- review 详情以现有 record 字段为准

这是对的。

但功能验收里仍然写着：

- 可以从任一 `replay / incident / review` 页面跳回关联 `snapshot` 或 `validation`

这对 `review` 页面来说仍然过强。

因为当前 `NarrativeQualityReviewRecord` 只稳定提供：

- `review_id`
- `source_artifact_reference`
- `evidence_references`

它并不稳定提供：

- snapshot id
- validation ref
- replay ref

因此，review 页面是否能“跳回关联 snapshot 或 validation”，在当前契约下最多只能是：

- 基于 `source_artifact_reference` 或 `evidence_references` 的 best-effort 导航

不能写成硬性保证。

影响：

- 验收口径会高于实际 contract 能力
- 前端实现时会被迫从 evidence 文本里猜引用关系
- 结果要么验收失败，要么形成隐性约定

修订建议：

- 把该条改成更精确的表述，例如：
  - replay / incident 页面应能跳回关联 snapshot 或 validation
  - review 页面若 `source_artifact_reference` 或 `evidence_references` 可解析，则应提供 best-effort 回链
- 或者如果要把 review 页回链写成硬性保证，就必须先扩展 review contract

## 建议给 Claude 的修订动作

建议本轮只要求 Claude 做以下三类修订，不必大改整体结构：

1. 补齐 `SreDataProvider` 对 `Artifact Browser` 和 `ReleaseAttemptRecord` 的覆盖
2. 给 `Artifact Browser` 增加 top-level artifact 与 bundle-derived section 的资源模型定义
3. 把 review 页回链从“硬保证”降级为“契约存在时保证，否则 best-effort”

## 最终口径

这版 V3 文档已经从“需要重定方向”进入了“需要补契约闭环”的阶段。

现在不建议再把上一轮已修复的问题翻回来，而是聚焦这 3 个剩余点：

- 抽象层是否自洽
- artifact 资源模型是否清晰
- 验收口径是否和现有 review contract 一致

这 3 点修掉之后，这份文档就基本可以作为下一轮前端实施输入。
