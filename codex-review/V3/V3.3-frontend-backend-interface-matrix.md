# V3.3 前后端接口对齐矩阵

## 1. 联调结论

当前前端和后端只做到了“provider 形状层”的部分对齐，还没有做到 repo-native artifact 契约完全对齐。

当前仓库的真实后端面只有两层：

- Python 控制面对象：`ReplayBundle`、`IncidentReport`、`ReleaseAttemptRecord`、`NarrativeQualityReviewRecord`
- `JsonArtifactStore` 按相对路径读写原始 JSON artifact

当前仓库**没有**这些天然存在的后端对象：

- HTTP API
- 原生 artifact list/index 接口
- 原生 snapshot artifact
- 原生 replay list / validation list / metrics summary / overview summary 对象
- 带时间元数据的 review list 对象

因此，V3.3 之后的前端 PR 不能再把以下内容写成“后端理应支持”：

- 通用 `updatedAt`
- 通用 `status` / `timeRange` 过滤
- 原生 `replay_id`
- 原生 `validation` 聚合列表
- 原生 `overview` / `metrics` 全局摘要

## 2. 术语

- `CAT`：由 repo-native artifact 目录生成的 catalog/index。它不是后端原生业务对象，但可以是 V3.3 `P0-1` 的正式中间产物。
- `RB`：`ReplayBundle`
- `IR`：`IncidentReport`
- `RAR`：`ReleaseAttemptRecord`
- `QRR`：`NarrativeQualityReviewRecord`

## 5. 绑定规则

自 V3.3 起，前端 PR 需求、页面文案、provider contract、测试断言必须服从本矩阵。

强制规则如下：

1. 不允许再写“后端理应支持”。
2. 任何字段若不是 `RB / IR / RAR / QRR` 的原生字段，就必须明确标成 `provider 推导` 或 `CAT 元数据`。
3. `snapshotRef`、`replayRef`、`validationRef`、`gateRef` 这类页面路由键，默认都视为前端/provider 合成 ref，除非后端对象本身定义了稳定主键。
4. `updatedAt` 不是通用原生字段。除非明确写成 `catalog time` 或 `filesystem mtime best-effort`，否则不能进入 P0 字段承诺。
5. `Validation`、`Metrics`、`Overview` 都是聚合面，不是后端原生 artifact；PR 文档里必须写清其聚合来源。
6. `Review` 页面不能再承诺时间筛选、状态筛选、稳定 snapshot 回链、稳定 validation 回链，除非先扩展 `QRR` 契约。
7. `Replay` 页面不能再承诺后端原生 `replay_id`；P0 路由必须以 bundle artifact ref 为主键语义。
8. `Metrics` 页面不能把全局聚合指标伪装成单 artifact 局部指标。

## 6. 对 V3.3 PR 的直接约束

V3.3 PR 需求必须据此收口成三条：

- `P0-1`：补 `CAT`，把 artifact mode 改成 repo-native artifact + catalog 驱动，而不是前端自带 fixture 驱动。
- `P0-2`：所有 query cache 以 `dataMode` 分区，保证 mock / artifact 不串缓存。
- `P0-3`：Validation Center 不得静默截断；要么 provider 补聚合接口，要么页面遍历全量 `RB`。

除此之外，前端 PR 文档还必须同步做三件事：

- 把每个页面的字段承诺改成“当前能保证”的字段集合
- 把每个派生面写清楚来源对象
- 把每个降级项写成显式限制，而不是留成模糊 TODO
