# alpha-SRE V2.2 PR Requirements

## 结论

V2.2 不是继续补 V2 核心链路，而是在 V2 已开发完成的基础上，把 narrative SRE control plane 收敛成可长期维护的稳定契约。

本 PR 的重点不再是“有没有 replay / gate / metrics / integration”，而是这些能力的边界是否已经闭合、证据是否可追溯、历史文档是否已经从现行需求源中剥离。

## 当前判断

- `V2-A` 到 `V2-F` 视为已完成的基线。
- `claude-review/V2/final-disposition.md` 仅保留历史 review 意义，不再作为现行需求源。
- `codex-review/v2-pr-requirements-narrative-kernel.md` 只保留 V2 起点背景，不再作为 V2.2 的实施路线图。
- 本 PR 只讨论现行契约的稳定化，不重新开启已关闭的 V2 blocker 列表。

## V2.2 目标

1. 补齐 snapshot 引用完整性和锁定语义。
2. 明确 schema evolution、artifact round-trip、integration write-back 的兼容行为。
3. 保证 replay、gate、metrics 继续按 narrative-native contract 工作。
4. 将历史 review 文档从现行需求判断中彻底分层出去。

## 需求范围

### 1. Snapshot 引用完整性必须闭合

#### 规范

- `NarrativeSnapshot` 中所有可解析引用都必须在 snapshot 内有确定落点，或者被显式标注为外部引用。
- 下列字段都必须参与结构化完整性校验：
  - `CharacterState.relationship_links`
  - `CharacterState.active_constraints`
  - `CharacterState.memory_references`
  - `CharacterState.belief_ids`
  - `CharacterState.capability_ids`
  - `RelationshipState.subject_character_id`
  - `RelationshipState.object_character_id`
  - `MemoryState.owning_character_id`
  - `ConstraintState.affected_actors`
  - `FactState.related_character_ids`
  - `FactState.related_rule_ids`
  - `BeliefState.holder_character_id`
  - `BeliefState.fact_id`
  - `BeliefState.derived_from_memory_ids`
  - `BeliefState.contradicts_fact_id`
  - `PlotThreadState.affected_characters`
  - `CapabilityState.character_id`
  - `CapabilityState.source_rule_id`
  - `CapabilityState.source_constraint_id`
  - `VisibilityEdgeState.fact_id`
  - `VisibilityEdgeState.viewer_id`
- 任何在 snapshot map 中无法定位的引用，都必须返回结构化 `ValidationIssue`，不得静默通过。

#### 解释

- 这不是“字段是否存在”的问题，而是“锁定状态是否自洽”的问题。
- 已定义为外部系统负责的引用，必须在 schema 中显式声明，不允许靠约定留空。

### 2. 锁定快照必须是可证据化的稳定 artifact

#### 规范

- validation 之后的快照不能因外部 caller 的原地修改而改变证据内容。
- `clone`、serialization、replay boundary 必须提供防御性复制，或等效的不可变容器。
- `NarrativeSnapshot` 的语义是“逻辑冻结”，不是“约定上不要改”。

#### 验收

- 任何在 validate 之后对 snapshot 容器的原地修改，都不能反向污染 replay、incident 或 drift evidence。
- snapshot 的序列化结果与 replay 输入必须保持一致可重建。

### 3. Schema evolution 必须成为现行契约的一部分

#### 规范

- 兼容规则采用同 major 版本兼容、structural additive 的 minor / patch 可读原则。
- 兼容性只约束 schema shape，不覆盖 policy version、visibility version 或 locked post-state 校验。
- 不兼容版本必须显式失败，不能 silent downgrade。

#### 需求

- read、write-back、replay、incident export 对兼容与不兼容版本的行为必须一致。
- 旧 bundle、旧 incident artifact、旧 replay bundle 的读取策略必须显式定义为以下三类之一：
  - 兼容读取
  - 显式迁移
  - 显式失败
- 不允许把历史 artifact 留在灰区。

### 4. Replay / gate / metrics 继续保持 narrative-native contract

#### 规范

- locked post-state verification 仍然是 replay 的硬约束。
- `ObservationFrame` 的 allowed / blocked / active rule 语义仍然是执行边界，而不是注释信息。
- `ReplayResult.ok == False` 默认不放行，gate 必须 fail closed，除非 issue 被明确归类为 warning。
- metrics 的分母继续使用 checked units，而不是总 issue 数。

#### 验收

- hidden fact leak、false belief conflict、capability violation、inactive rule use、plot obligation miss、post-state mismatch 都必须可区分。
- gate 不能因为 issue 名字不在白名单而误放行失败 replay。
- 任何新的 narrative failure class 都必须同步进入 metrics、gate 和 artifact 语义链。

### 5. 历史 review 文档必须退回历史层

#### 规范

- `claude-review/V2/final-disposition.md` 只作为历史 review disposition，不再作为现行需求源。
- `codex-review/v2-pr-requirements-narrative-kernel.md` 只作为 V2 背景，不再作为 V2.2 的需求合同。
- V2.2 的 PR 文档只写“当前要保证什么”，不把文件名、测试名、切片顺序写成 contract 本体。

#### 验收

- 需求文档和实施路线图可以分离阅读。
- 改文件布局不会构成假违约。
- 新需求不会被旧 blocker 列表拖回去。

## 非目标

- 不再重新定义 V2 kernel 字段。
- 不再把 replay、gate、metrics、integration 当成从零搭建。
- 不做 `alpha-autopilot` 的进一步集成扩张。
- 不把实施顺序写进规范正文。
- 不重新开启已关闭的 V2 P1 blocker 讨论。

## 建议验收清单

- snapshot 引用完整性覆盖所有 narrative surfaces。
- snapshot 在 validate 后不再被外部原地修改污染。
- 兼容 schema 版本的 read、replay、write-back 行为一致。
- 失败 replay 默认 fail closed。
- metrics 继续使用 checked denominator。
- 历史 disposition 文档不再参与现行需求判断。

## 非规范实现建议

- 优先补齐 snapshot referential validation 和 immutability boundary。
- 再收敛 schema evolution 和 artifact compatibility 的文档与测试。
- 最后补 drift 与 incident lineage 的表述一致性。

## 交付口径

V2.2 合并时，评审重点应是：

1. 现行契约是否稳定。
2. 历史文档是否已被隔离。
3. 失败语义是否仍然 fail closed。
4. artifact 是否可回放、可解释、可追踪。
