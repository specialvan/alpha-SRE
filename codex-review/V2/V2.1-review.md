# alpha-SRE V2.1 Review

## 结论

本轮 V2.1 review 的结论是：继续维持 request changes。

当前仓库不是空骨架。它已经具备可执行的 V1 event-centric control loop，包括 snapshot validation、replay、causal validation、metrics、gate、incident artifact、integration bridge 和测试覆盖。

但 V2/V2.1 的合并门槛不能只看“链路能跑”。系统必须证明它能作为 narrative SRE control plane 发挥权威作用：锁定状态可复盘，终态可对账，观察框架可执行，gate 不漏放失败，状态引用完整且可作为可靠证据。

当前代码仍未达到这个门槛。

## 与 Claude V2 评审包的对齐

Claude V2 评审包最终已经和 Codex counter-review 收敛到一个正确前提：

> 当前仓库是一个 working V1 baseline，不是 hollow scaffold；但它还没有升级成 narrative-state-centric SRE kernel。

这个前提仍然成立。V2.1 review 不再纠缠“V1 是否落地”，而是把焦点推进到：

1. V1 control loop 哪些地方会破坏 SRE gate authority。
2. 已新增的 V2-A state kernel 是否真正进入 replay / causal / metrics / gate 判定路径。
3. 哪些问题必须在 V2.1 合并前修复，哪些可以进入后续 hardening。

## V2.1 新增确认 findings

### F1. `post_state_snapshot` 没有被对账

位置：`alpha_sre/replay.py:423-430`

`ReplaySession` 接受 `post_state_snapshot`，但 `replay_session()` 没有把它传入 replay，也没有比较 replay 后的 `ReplayResult.state` 与 locked post-state。

我用目标终态和实际 replay 终态故意不一致的 session 验证过：结果仍为 `ok=True`，且没有 issue。

影响：

- V1 只能证明 event chain 被结构性重放。
- 不能证明 locked run 的终态被复现。
- V2 semantic replay 的核心承诺不成立。

合并要求：

- `ReplaySession.post_state_snapshot` 必须进入 replay 判定路径。
- replay 结果必须产生稳定的 post-state diff。
- mismatch 必须产生 `post_state_mismatch` 或同等级 narrative failure class。
- gate 必须阻断该 failure。

### F2. Gate 可以放行失败 replay

位置：`alpha_sre/gate.py:66-67`

`ConsistencyGate.evaluate()` 只把 `CRITICAL_ISSUE_CODES` 或 `_mismatch` 后缀 issue 变成 blocking issue。任何未列入白名单的新 validation issue 都可能导致：

- `ReplayResult.ok == False`
- `GateResult.allowed == True`

我验证过 `missing_snapshot_id` 会出现这种漏放。

影响：

- gate authority 被破坏。
- 新增 V2 kernel issue 例如 `dangling_belief_fact`、`dangling_visibility_viewer`、`invalid_plot_obligation_state` 可能在未加入白名单时被漏放。
- 后续扩展 narrative failure class 时容易出现默认放行。

合并要求：

- 对 `replay.ok == False` 的默认策略应是 fail closed。
- 如果确实存在 non-blocking issue，必须显式降级为 warning class，而不是依赖不在白名单中。
- `CRITICAL_ISSUE_CODES` 应从 allow-by-omission 改成 block-by-default with explicit warning exceptions。

### F3. `add_memory` / `world_rule_update` payload 会 KeyError

位置：`alpha_sre/replay.py:307-340`

`add_memory` 分支验证的是 `payload["memory"]` 里的 `MemoryState.memory_id`，但写入时读取 `payload["memory_id"]`。如果事件只携带 `memory` 对象而没有冗余顶层 `memory_id`，replay 会抛 `KeyError`，而不是返回结构化 `ValidationIssue`。

`world_rule_update` 对 `rule_id` 也有同类问题。

影响：

- replay 不是 total function。
- artifact replay / incident replay 可能因 payload 形态差异直接崩溃。
- integration bridge 不能稳定把失败包装成 gate / incident evidence。

合并要求：

- 以 payload 内对象 ID 为权威来源，或显式校验顶层 ID 与对象 ID 一致。
- 缺失 / 不一致必须返回 `invalid_memory_payload`、`memory_payload_id_mismatch`、`invalid_world_rule_payload` 或同级结构化 issue。
- 添加 regression tests 覆盖 payload 缺字段、不一致、对象缺 ID 三类情况。

### F4. ObservationFrame 的 allowed list 被忽略

位置：`alpha_sre/replay.py:267-268`

replay 当前只检查 `event.event_type in blocked_event_types`，没有检查 `allowed_event_types`。因此 observation frame 声明“只允许 update_goal”时，`chapter_outcome` 只要通过其他因果校验仍可执行。

`active_world_rule_ids` 也没有进入 replay 判定路径。

影响：

- observation frame 仍主要是 metadata，不是 executable boundary。
- capability / action window 不能成为 SRE 级控制。
- hidden fact / belief / rule activation 的 V2 语义无法闭环。

合并要求：

- 当 `allowed_event_types` 非空时，非 allowed event 必须产生 `impossible_action` 或更精确的 `event_type_not_allowed`。
- `blocked_event_types` 与 `allowed_event_types` 的冲突仍应保留 contract validation。
- `active_world_rule_ids` 至少要参与 inactive-rule use 检查，后续再和 `WorldRuleState.activation_status` 及 `CapabilityState` 对齐。

### F5. Snapshot 引用完整性不足

位置：`alpha_sre/state.py:323-334`

`CharacterState.relationship_links`、`active_constraints`、`memory_references` 没有被检查是否存在于 snapshot map 中。我验证过角色引用缺失 memory、constraint、relationship 时，`snapshot.validate()` 仍返回 `ok=True`。

影响：

- locked snapshot 可能包含悬空叙事引用。
- memory omission / relationship state / constraint enforcement 的后续判断会建立在不完整状态上。
- V2 kernel 虽然新增了 belief/capability 等 dangling checks，但 V1 原有引用面仍有缺口。

合并要求：

- `CharacterState.memory_references` 必须指向 `NarrativeSnapshot.memories`。
- `CharacterState.active_constraints` 必须指向 `NarrativeSnapshot.constraints`。
- `CharacterState.relationship_links` 必须有稳定 key 规则并指向 `NarrativeSnapshot.relationships`。
- 为三类 dangling reference 增加 failure-class-first tests。

### F6. Locked state 仍可被调用方原地修改

位置：`alpha_sre/state.py:286-304`

dataclass 使用了 `frozen=True`，但 snapshot 内部仍是 mutable dict，子对象内部仍有 mutable list。调用方可以在 validate 之后继续修改：

- `snapshot.characters["c2"] = ...`
- `snapshot.characters["c1"].memory_references.append(...)`

影响：

- locked snapshot 不是实际不可变 artifact。
- replay evidence 可能在校验后被外部突变。
- artifact、incident、drift 报告中的证据稳定性受损。

合并要求：

- 明确状态层的 immutability 策略。
- 至少在 clone / serialization / replay boundary 做 deep copy 与 tuple 化，避免外部 list/dict 被共享。
- 如果短期不改底层容器，也必须在文档中标注当前 snapshot 是 conventionally locked，而不是 physically immutable，并补 mutation regression tests 或防御性 copy。

## V2.1 合并 blockers

### B1. Replay 必须执行 locked post-state verification

这是 V2.1 的最高优先级 blocker。没有终态对账，就没有 semantic replay。

最低验收：

- `ReplaySession.post_state_snapshot` 被消费。
- replay result 包含 post-state diff。
- mismatch 会让 `ReplayResult.ok == False`。
- gate 阻断 mismatch。
- incident report 能保留 mismatch evidence reference。

### B2. Gate 必须 fail closed

gate 不应只阻断白名单 issue。对失败 replay 默认放行会破坏整个 SRE 控制面。

最低验收：

- 任意 `ReplayResult.ok == False` 默认 `GateResult.allowed == False`。
- 非阻断问题必须有显式 warning taxonomy。
- 所有新增 V2 narrative issue 都有确定 hard / soft 处置。

### B3. Replay payload handling 必须结构化失败

replay 不能因输入 payload 差异直接抛 KeyError。

最低验收：

- `add_memory`、`world_rule_update` 缺 ID 或 ID 不一致时返回 ValidationIssue。
- integration write-back 不会因该类 payload 崩溃。
- artifact round-trip 后 replay 行为一致。

### B4. ObservationFrame 必须成为 replay 约束

最低验收：

- `allowed_event_types` 非空时被强制执行。
- `active_world_rule_ids` 至少覆盖 inactive-rule use 的第一版检查。
- visible / hidden / believed facts 的检查进入 semantic replay slice。

### B5. Snapshot state integrity 必须补齐

最低验收：

- 原有 V1 引用面和新增 V2-A 引用面都能被 `NarrativeSnapshot.validate()` 覆盖。
- dangling references 都会导致 replay preflight failure。
- gate 不会漏放这些 failure。

### B6. Metrics / causal taxonomy 继续作为 V2.1 阻断项

这部分继承 Claude V2 评审结论。即使 P0 state fields 已经部分落地，metrics 和 causal validation 仍需迁移到 narrative-native checked units 和 failure classes。

最低验收：

- `causality_break_rate` 不再使用 total issue count 作为核心分母。
- `visibility_leak_rate` 迁移到 checked visibility decisions。
- causal findings 能表达 belief conflict、capability violation、inactive rule misuse、plot obligation miss、post-state mismatch。

## 非阻断但应跟踪的 hardening

### H1. Integration drift surface 扩大

`IntegrationBridge.build_drift_report()` 当前只比较 state identity、schema、policy、visibility version。它不应该抢在 semantic replay 前成为最高优先级，但在 post-state verification 完成后，drift signature 应扩展到 narrative surfaces。

### H2. Incident lineage 细化

incident export 目前只能继承 replay / causal 输出的语义质量。等 B1-B6 落地后，应把 semantic failure class、post-state diff、observation frame evidence 纳入 incident lineage。

### H3. Schema evolution policy

V2-A state kernel 已经开始扩展 schema。后续需要明确旧 artifact 读取策略、缺省策略、显式失败策略和 migration policy。

## 推荐测试资产

新增或扩展以下 golden / regression tests：

- locked post-state mismatch is rejected
- replay session post-state match is accepted
- gate blocks failed replay even for unknown validation issue
- gate downgrades only explicit warning classes
- add_memory payload object-only form does not crash
- add_memory top-level memory_id mismatch is rejected
- world_rule_update rule_id mismatch is rejected
- observation allowed_event_types rejects non-allowed events
- active_world_rule_ids rejects inactive rule usage
- character dangling memory reference is rejected
- character dangling constraint reference is rejected
- character dangling relationship link is rejected
- snapshot mutation after lock does not affect replay evidence
- metrics use checked outcomes / visibility decisions rather than total issue count

## 最终评审口径

V2.1 应被视为对 Claude V2 评审包的工程化收敛版：

- 接受 Claude 对 semantic gap 的主判断。
- 修正 Claude 早期“V1 skeleton”表述：当前系统是可执行 V1 baseline。
- 更新 Claude 对 state kernel 的状态判断：字段层面已有 V2-A 进展，但 semantic replay / gate / metrics 仍未完成。
- 新增 V2.1 级别的 gate authority、payload totality、snapshot integrity 和 locked-state immutability blockers。

在 B1-B6 完成前，不建议合并 V2.1 为“完成的 narrative SRE baseline”。可以接受分 slice 合入，但每个 slice 必须带 failure-class-first tests 和对应规格更新。
